A data model is ==a structured blueprint or abstract framework that defines how data is organized, stored, and accessed within a database==. It defines data elements, their attributes, and relationships to ensure data integrity and consistency. Common types include conceptual, logical, and physical models, often visualized as diagrams to guide database design.

# ER Model (Entity-Relationship Model)
The Entity-Relationship Model ([ER Model](https://www.geeksforgeeks.org/dbms/introduction-of-er-model/)) is a conceptual model for designing a database. This model represents the logical structure of a database, including entities, their attributes, and relationships between them.

![[ER_Diagram.png]]

Here are the three core building blocks of an ER model:

###### 1. Entities (The Nouns)
An entity is any distinct, real-world object or concept that you need to store data about. If you were building a system for a university, your entities would be the core subjects of that system.

- **Examples:** `Student`, `Professor`, `Course`.
- _Visually:_ Entities are usually drawn as **rectangles**.

###### 2. Attributes (The Adjectives)
Attributes are the specific pieces of information that describe an entity.

- **Examples:** For a `Student` entity, the attributes might be `Student_ID`, `First_Name`, `Last_Name`, and `Date_of_Birth`.
- _Visually:_ Attributes are usually drawn as **ovals** connected to their parent entity.

###### 3. Relationships (The Verbs)
This is where the ER model gets its name. Relationships describe exactly how two entities interact with each other.

- **Examples:** A Student **enrolls in** a Course. A Professor **teaches** a Course.
- _Visually:_ Relationships are usually drawn as **diamonds** connecting two distinct entities.

# Relational Model
**Relational Model** is the actual, structural framework you build inside the computer. It is the most common way data has been organized and stored since the 1980s. When you hear people talk about "SQL databases" (like MySQL, PostgreSQL, or SQLite), they are talking about databases built on the Relational Model.

Here is how the terminology translates from the drawing board to the actual database:
###### 1. Tables (The Entities)
In a relational database, every distinct concept gets its own table. If you are building an app, you might have a `Users` table, a `Products` table, and an `Orders` table.

- _Formal math term:_ Relation.

###### 2. Columns (The Attributes)
The columns define the specific pieces of information every entry in the table must have. In the `Users` table, your columns might be `ID`, `Username`, `Email`, and `Signup_Date`. The database is incredibly strict about these: if a column is set up to hold a date, you cannot put text in it.

###### 3. Rows (The Records)
Every time a new user signs up, a new row is added to the `Users` table. A row represents one single, complete instance of data.

- _Formal math term:_ Tuple.

# Hierarchical Model
While the Relational Model is the modern standard for general-purpose databases, the **Hierarchical Model** is one of the oldest ways to organize data. It was actually developed by IBM in the 1960s to manage the massive amounts of manufacturing data needed for the Apollo space program!
If the relational model is a collection of connected spreadsheets, the hierarchical model is an **upside-down tree**.
Here is how it works and why we still see its influence today:

**The Tree Structure**
In this model, data is organized into a strict "parent-child" relationship.

1. **The Root:** Everything starts at a single point at the top, called the root node.

2. **Parents and Children:** From the root, data branches downwards. A parent record can have many child records, but **a child record can only have ONE parent**.

Think of the file system on your computer. You have a main hard drive (the root). Inside it, you have a "Documents" folder and a "Photos" folder (children of the root). Inside the "Photos" folder, you might have "2023" and "2024" folders. The "2023" folder cannot simultaneously live inside both the "Photos" folder and the "Documents" folder. It has one strict path.
