In a live database, thousands of people might be reading, writing, and updating data at the exact same millisecond. To prevent chaos, database management systems bundle operations into single units of work called **transactions**.
**[ACID](https://www.geeksforgeeks.org/dbms/acid-properties-in-dbms/)** is an acronym representing the four mandatory properties a transaction must possess to guarantee that the database remains accurate and reliable, even in the event of a total system crash or power failure.

To understand them, let's use the classic example of a bank transfer: You are transferring $100 from Account A to Account B. This transaction actually requires two separate database steps:

1. Subtract $100 from Account A.
2. Add $100 to Account B.

#### 1. Atomicity ("All or Nothing")
Atomicity guarantees that a transaction is treated as a single, indivisible unit. It cannot be partially completed.

- **The Problem:** What if the database successfully subtracts $100 from Account A, but right before it adds the money to Account B, the server loses power? The $100 would just vanish into the digital void.

- **The ACID Solution:** Atomicity ensures that if _any_ step of the transaction fails, the entire transaction is instantly aborted and **rolled back** to its original state. The money stays in Account A. It is all or nothing.

#### 2. Consistency ("The Rules are the Rules")
Consistency guarantees that a transaction can only bring the database from one valid, legal state to another valid, legal state.

- **The Problem:** What if you try to transfer money to a bank account number that doesn't exist? Or what if a developer writes code that tries to insert text into a database column strictly meant for dates?

- **The ACID Solution:** Remember the rules we set up during database design (Primary Keys, Foreign Keys, Data Types, and Unique constraints)? Consistency enforces them ruthlessly. If a transaction tries to break any established database rule or constraint, it is immediately rejected and rolled back.

#### 3. Isolation ("Stay in Your Lane")
Isolation guarantees that when multiple transactions are happening at the exact same time, they do not interfere with one another. Each transaction must execute as if it were the only one running on the system.

- **The Problem:** Imagine a concert has exactly one ticket left. You and another person click "Buy" at the exact same millisecond. Without isolation, both of your transactions might read that there is "1" ticket available, process your payments, and sell the same ticket twice.

- **The ACID Solution:** Isolation effectively puts blinders on concurrent transactions. It locks the specific data being used so that one transaction must finish before the other is allowed to look at it. One of you gets the ticket; the other gets an "Out of Stock" error.

#### 4. Durability ("Written in Stone")
Durability guarantees that once a transaction is completed and the database confirms "Success," that change is permanent, no matter what happens next.

- **The Problem:** A database does a lot of its high-speed thinking in temporary memory (RAM). If it confirms your $100 transfer but only holds that information in RAM, a sudden server crash would wipe your transaction from existence.

- **The ACID Solution:** Durability ensures that the database permanently writes the results of a transaction to non-volatile storage (like a hard drive) _before_ it tells the user the transaction was successful. If someone pulls the power cord out of the server one second after you get your confirmation, your money will still be exactly where it belongs when the system boots back up.

#### Example:
```
-- Complete Bank Transfer with ACID
START TRANSACTION;

-- Check Atomicity: Both must succeed
  UPDATE accounts
  SET balance = balance - 1000
  WHERE account_id = 'A001' AND balance >= 1000; -- Consistency check
  
  -- Verify rows affected (Consistency)
  -- If 0 rows affected, balance was insufficient
  
  UPDATE accounts
  SET balance = balance + 1000
  WHERE account_id = 'A002';
  
  -- Log for Durability
  INSERT INTO transaction_log
  VALUES (NOW(), 'A001', 'A002', 1000, 'TRANSFER');
  
COMMIT;     -- Durable: persisted to disk
            -- Isolated: other transactions see consistent view
```
